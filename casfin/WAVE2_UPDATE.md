# CasFin — Wave 2 Update
Wave 1 shipped the encrypted resolution flow, CoFHE coprocessor integration, keeper bot spec, and the core game contracts. We also received direct feedback from reviewers on wallet UX, SDK completeness, and visual polish. Wave 2 addressed every piece of that feedback and pushed the platform significantly forward. This update covers what we shipped, what we rebuilt, and what's next.

## Full CoFHE SDK Integration
The biggest engineering push this wave was completing the `@cofhe/sdk` integration end-to-end. In Wave 1, the encryption path was partially wired — client-side encryption worked but the resolution flow still had gaps. Wave 2 rebuilt the entire pipeline.
**Client-side encryption.** Every bet input is now encrypted in the browser before reaching the network. For Coin Flip, that's `encryptBool(guessHeads)` for your side selection and `encryptUint128(amountWei)` for bet size. For Dice, `encryptUint8(targetNumber)` plus `encryptUint128(amount)`. The CoFHE session initializes automatically on wallet connection — the frontend tracks `sessionReady`, `sessionInitializing`, and `cofheConnected` states and disables the bet button until the TFHE engine is fully warmed. No user action required.
**Encrypted resolution flow.** Resolution runs in two async transactions by design. The first submits a decrypt task to the CoFHE network via the `TaskManager` precompile. The second — called after the coprocessor completes — reads the decrypted result and settles the vault balance. The keeper bot bridges this async gap automatically. Both the encrypted max-bet cap validation and the win/loss flag comparison happen entirely inside FHE — the contract never sees plaintext values at any point.
**`forceResolve` for testnet.** On Arbitrum Sepolia, the CoFHE coprocessor nodes aren't present (they only run on Fhenix-native chains). We added a `forceResolve` function to each game contract that bypasses the async decrypt path and settles bets using on-chain randomness from `block.prevrandao`. This is testnet-only — when CasFin deploys on Fhenix infrastructure, `forceResolve` gets disabled and the full FHE async flow takes over. Same resolution interface, different randomness source. Zero frontend changes needed for the migration.

## Privy Wallet Integration
This was direct feedback from Wave 1 reviewers: the wallet connection UX was raw MetaMask-only, requiring users to manually switch networks and handle every RPC error themselves. Wave 2 replaced the entire wallet layer with Privy.
**What Privy gives us.** Embedded wallets for users who don't have MetaMask. Social login (email, Google, Twitter) as an onramp — users can start playing without ever installing a browser extension. Automatic network switching to Arbitrum Sepolia. Session persistence across page refreshes. The Privy provider wraps the entire app in `layout.tsx`, and every component accesses wallet state through our `useWallet()` hook which abstracts Privy's internals.
**Why it matters for a casino.** Onboarding friction kills conversion. A user who has to install MetaMask, create a wallet, get testnet ETH, and manually add Arbitrum Sepolia to their network config will leave before placing a single bet. Privy reduces that to: click "Connect" → sign in with Google → play. The embedded wallet handles chain config, gas estimation, and transaction signing transparently.

## New Game: Poker
Wave 2 added a full video poker implementation to the casino. Four games now live on the platform: Coin Flip, Dice, Crash, and Poker. Poker uses the same FHE encryption pipeline — card values are encrypted on-chain, the hand evaluation runs over encrypted state, and payouts settle to the encrypted vault balance. The `CleanPokerCard` component follows the same architecture as the other game cards: CoFHE session gating, encrypted input submission, keeper-driven async resolution, and outcome display via `CasinoOutcomeCard`. The poker contract (`EncryptedPokerGame.sol`) is deployed alongside the other three game contracts and managed by the same keeper bot.

## Multiple RPC Endpoints for Load Balancing
Reliability was an issue in Wave 1. A single Infura endpoint handling all frontend reads, keeper transactions, and CoFHE session initialization meant that rate limits and transient failures would cascade across the entire platform. Wave 2 introduced multi-RPC load balancing.
The frontend configuration now supports multiple RPC URLs (`NEXT_PUBLIC_RPC_URL_1` through `NEXT_PUBLIC_RPC_URL_4`). The provider layer rotates between endpoints on failure and distributes read calls across available RPCs. The keeper bot has its own dedicated RPC endpoint (`KEEPER_RPC_URL`) separate from the frontend to prevent user-facing reads from competing with resolution transactions for rate-limit headroom. Infura, Alchemy, and public Arbitrum Sepolia endpoints are all configured as fallbacks. The result: no single RPC failure takes down the platform.

## Casino UI/UX Overhaul
The casino page was rebuilt from scratch into a terminal-grade trading interface. Every visual element was redesigned.
**Typography.** JetBrains Mono (weight 300) for all UI text — monospace, terminal-native. DM Serif Display for display headings. All labels uppercase, 10-11px, letter-spacing 0.15-0.2em. Both loaded via `next/font/google` with CSS variable binding (`--font-mono`, `--font-serif`).
**Color system.** Single accent: warm gold `#f5c842` against deep space navy `#0a0e1a`. No secondary colors, no gradient soup. Gold borders at 20% opacity on hover, gold box-shadows at 4% for ambient glow. Entire palette in CSS custom properties (`--terminal-gold`, `--terminal-bg`).
**Glass panels.** Game cards use `rgba(12, 17, 32, 0.75)` with `backdrop-filter: blur(20px)`. Gold corner accents via pseudo-elements. Hover shifts border to gold with subtle ambient glow.
**Animations.** CTA buttons pulse gold (3s ease-in-out infinite). Scanline overlay across the viewport via repeating-linear-gradient at 3px intervals. Encrypted values use a glitch-text animation (translate + skew + text-shadow displacement, 5s loop). Ticker tape marquee scrolls live protocol state (vault TVL, crash status, FHE engine status) continuously.
**Layout.** Centered single-column at 700px max-width replacing the old sidebar + two-column grid. Tab switcher converted to pill buttons — gold active state with glowing box-shadow, gray inactive. Fhenix Engine badge with lock icon and "On-Chain Encrypted State" label. Bottom dock concept for vault quick-actions. The result: a casino that reads like a Bloomberg terminal, not a hackathon prototype.

## Keeper Bot Infrastructure
The keeper was a local-only script — close your laptop, keeper stops, bets hang. We built two serverless deployment paths.
**AWS Lambda + EventBridge.** Keeper logic extracted into a stateless `runKeeperTick()` function. Lambda executes one tick per invocation. EventBridge triggers every 60 seconds. Cost: $0 under AWS free tier. Serverless Framework handles deployment — single command creates Lambda, EventBridge rule, and CloudWatch logs.
**Vercel Cron.** Keeper runs as a Next.js API route (`/api/keeper/tick`) secured by `CRON_SECRET`. Works for teams already on Vercel. Pro plan required for sub-hourly intervals.

## Security Hardening
GitGuardian flagged exposed credentials in committed documentation. Three secrets found: Neon PostgreSQL URI with password and keeper private key (twice). All redacted, git history rewritten via soft reset and force-push. Old commits purged from remote. All credentials flagged for rotation. The `.env` files were already gitignored — the leak came from docs containing copy-pasted config examples. Lesson: secrets leak through documentation, not just env files.

## Wave 3: Event-Driven Resolution (WebSocket + Redis)
Polling is dead architecture. Wave 3 replaces it with WebSocket event subscription + Redis pub/sub for instant bet resolution.
**WebSocket.** Keeper connects via `wss://` and subscribes to contract events (`BetPlaced`, `RoundCreated`, `ClaimRequested`). Hears events within the same block — zero polling delay. Immediately submits resolution. Gap drops from 30-90 seconds to 3-6 seconds.
**Redis pub/sub.** Keeper publishes resolution results to Redis. Frontend subscribes via Server-Sent Events through `/api/events/bets`. UI updates live — no refresh, no polling. Provider: Upstash (serverless, free tier).
**Full pipeline.** Arbitrum WebSocket → Keeper listener → resolve tx → Redis publish → Frontend SSE → live UI. Backup polling worker catches missed events during reconnection. This is production architecture — same pattern as Polymarket and Rollbit.

## The Position
Every on-chain casino publishes your entire gambling history as a public spreadsheet. Bet amounts, guesses, wins, losses, strategy patterns — permanently readable by anyone with a block explorer. They call it "provably fair" and stop there.
CasFin uses the same verifiable randomness (Chainlink VRF), same on-chain settlement, same auditable contracts. But we encrypt the entire game state end-to-end using Fhenix CoFHE. Your bet enters as ciphertext. The outcome comparison happens homomorphically. The payout adjusts your encrypted balance. The only moment anything is readable is inside your browser, decrypted with your session key. Not the house, not other players, not MEV bots, not validators — no one sees your bet.
Wave 1 built the cryptography. Wave 2 made the infrastructure production-ready and addressed every piece of reviewer feedback. Wave 3 makes resolution instant.
